Pulse

The entire API is 3 methods: event, emit, on. Events flow through a DAG with deterministic propagation. Works for UI state, 60fps animation, and async. Everything else is a pattern.

3 Methods DAG Propagation Glitch-Free Join Patterns Zero Config Framework Agnostic
$ npm install @pulse/core
Get Started API Reference GitHub

Key Concepts

Persistent, Ref-Counted Events

Events are not fire-and-forget. When emitted, an event is deposited into a mailbox and remains there until every registered handler has processed it. Each event tracks pending consumers via a ref-count, and is recycled to an object pool only after all consumers finish.

Synchronous DAG Propagation

Handlers form a directed acyclic graph. On emit, Pulse computes a topological ordering and drains mailboxes in order. Downstream handlers always see upstream events in the same cycle -- no glitches, no stale reads, fully deterministic.

Static DAG Declaration

The DAG is declared statically, not discovered at runtime. Chaining (on(A).emit(B)) and output arrays (on(A, [B, C], handler)) declare edges at registration time. No runtime tracing is needed -- the full graph is known before any events fire. This enables ahead-of-time visualization, validation, and cycle detection.

Join Patterns

Pass an array of event types to on() and the handler fires only when all of them have pending events. This is barrier synchronization built into the API.

Quick Start

Install

bash 
npm install @pulse/core

The Entire API

typescript 
import { createEngine, Skip } from '@pulse/core'

const engine = createEngine()

// Declare events
const Click = engine.event<{ x: number; y: number }>('Click')
const Highlight = engine.event<string>('Highlight')

// React to clicks by emitting highlights
engine.on(Click, (payload) => {
  engine.emit(Highlight, `Clicked at (${payload.x}, ${payload.y})`)
})

// Side effect: log highlights
engine.on(Highlight, (message) => {
  console.log(message)
})

// Fire
engine.emit(Click, { x: 100, y: 200 })
// logs: "Clicked at (100, 200)"

// Cleanup
engine.destroy()

API Reference

engine.event<T>(name)

Declare a named event type. The type parameter T defines the payload shape. Event types are channels -- declarations, not instances.

typescript 
event<T = void>(name: string): EventType<T>
typescript 
// Void payload (no data)
const Reset = engine.event('Reset')

// Typed payload
const UserLogin = engine.event<{ userId: string; timestamp: number }>('UserLogin')

// Generic payload
const Message = engine.event<string>('Message')

The engine appends a unique counter to the name (e.g., 'Click' becomes 'Click#0'), so two calls to engine.event('Click') produce distinct types.

engine.emit(type, payload)

Emit an event. Deposits the event into the mailbox, triggers synchronous DAG propagation, and drains any pending emissions.

typescript 
emit<T>(type: EventType<T>, payload: T): void
typescript 
const Increment = engine.event<number>('Increment')

engine.emit(Increment, 1)
engine.emit(Increment, 5)

If emit is called during propagation (e.g., from inside a handler), the emission is queued and processed after the current propagation cycle completes.

engine.on(type, handler)

Subscribe to events. The handler runs as a side effect and can call engine.emit() inside to chain events through the DAG. Returns an unsubscribe function.

typescript 
on<T>(type: EventType<T>, handler: (payload: T) => void): () => void
on(types: EventType[], handler: (...payloads: any[]) => void): () => void

Side Effect

typescript 
const Notification = engine.event<string>('Notification')

const unsub = engine.on(Notification, (message) => {
  console.log('Notification:', message)
})

engine.emit(Notification, 'Hello!')
// logs: "Notification: Hello!"

unsub() // stop listening

Chaining (emit inside a handler)

typescript 
const RawTemp = engine.event<number>('RawTemp')
const Celsius = engine.event<number>('Celsius')

engine.on(RawTemp, (kelvin) => {
  engine.emit(Celsius, kelvin - 273.15)
})

Async (await then emit)

typescript 
const SearchQuery = engine.event<string>('SearchQuery')
const SearchLoading = engine.event('SearchLoading')
const SearchDone = engine.event<string[]>('SearchDone')
const SearchError = engine.event<Error>('SearchError')

engine.on(SearchQuery, async (query) => {
  engine.emit(SearchLoading, undefined)
  try {
    const res = await fetch(`/api/search?q=${query}`)
    const results = await res.json()
    engine.emit(SearchDone, results)
  } catch (err) {
    engine.emit(SearchError, err as Error)
  }
})

Join (fires when all input types have events)

typescript 
const Username = engine.event<string>('Username')
const Password = engine.event<string>('Password')
const LoginAttempt = engine.event<{ user: string; pass: string }>('LoginAttempt')

engine.on([Username, Password], (user, pass) => {
  engine.emit(LoginAttempt, { user, pass })
})

engine.emit(Username, 'alice')
// LoginAttempt does NOT fire yet -- waiting for Password
engine.emit(Password, 's3cret')
// LoginAttempt fires with { user: 'alice', pass: 's3cret' }

Skip

A sentinel symbol exported from @pulse/core. Return it from a handler to indicate that no downstream emission should occur. Useful for conditional logic where you want to explicitly signal "nothing happened."

typescript 
import { createEngine, Skip } from '@pulse/core'

const engine = createEngine()

const Input = engine.event<number>('Input')
const PositiveOnly = engine.event<number>('PositiveOnly')

engine.on(Input, (value) => {
  if (value <= 0) return Skip
  engine.emit(PositiveOnly, value)
})

engine.destroy()

Full teardown. Removes all handlers, clears all mailboxes, stops all timers, and releases all resources.

typescript 
engine.destroy()
// Engine is now inert. All subscriptions are cleaned up.

Static DAG

Every form of on() declares its DAG edges at registration time. The graph is fully known before any events fire -- no runtime tracing required.

How each on() form contributes to the DAG

DAG edges 
engine.on(A).emit(B)           // edge: A → on(A) → B
engine.on(A, [B, C], handler)  // edges: A → on(A) → B, A → on(A) → C
engine.on(A, handler)          // terminal: A → on(A), no outgoing edges

Chaining (on(A).emit(B)) and output arrays (on(A, [B, C], handler)) both declare edges statically. A plain handler with no declared outputs is a terminal node in the graph.

Introspecting the DAG

Call engine.getDAG() to retrieve the full graph at any time. The returned object contains all registered rules (nodes) and the edges between them.

typescript 
const dag = engine.getDAG()
// dag.nodes: Rule[]          -- every registered handler/chain
// dag.edges: [Rule, Rule][]  -- directed edges between rules

Because the DAG is static, you can visualize or validate it before emitting a single event -- useful for build-time checks, documentation generation, and DevTools integration.

Example: Generated DAG

Here's a real music player built with Pulse. The code on the left produces the DAG on the right — all edges are known at registration time, before any events fire.

engine.ts 
const Play = engine.event('Play')
const Pause = engine.event('Pause')
const NextTrack = engine.event('NextTrack')
const Seek = engine.event('Seek')
const Frame = engine.event('Frame')

// State events
const IsPlayingChanged = engine.event('IsPlayingChanged')
const TrackChanged = engine.event('TrackChanged')
const ProgressChanged = engine.event('ProgressChanged')
const VisualizerChanged = engine.event('VisualizerChanged')

// Chaining — static DAG edges
engine.on(Play, [IsPlayingChanged], (_, set) => {
  isPlaying = true; set(true)
})
engine.on(Pause, [IsPlayingChanged], (_, set) => {
  isPlaying = false; set(false)
})
engine.on(Seek, [ProgressChanged], (v, set) => {
  progress = v; set(v)
})
engine.on(NextTrack, [TrackChanged, ProgressChanged],
  (_, setTrack, setProgress) => {
    currentTrack = getNext()
    setTrack(currentTrack)
    setProgress(0)
  })
engine.on(Frame, [ProgressChanged, VisualizerChanged],
  (dt, setProgress, setViz) => {
    if (!isPlaying) return
    progress += dt / duration
    setProgress(progress)
    setViz(generateBars())
  })

Every on() call with an outputs array creates edges in the DAG. Call engine.getDAG() to retrieve the full graph — nodes are rules, edges are the data flow connections. All edges exist before any emit().

typescript 
const dag = engine.getDAG()
console.log(dag.nodes.length)  // 5 rules
console.log(dag.edges.length)  // 7 edges
// Edges: Play→IsPlayingChanged, Pause→IsPlayingChanged,
//        Seek→ProgressChanged, NextTrack→TrackChanged,
//        NextTrack→ProgressChanged, Frame→ProgressChanged,
//        Frame→VisualizerChanged

Chaining Syntax

The on(A).emit(B) chaining syntax is a declarative shorthand for the common pattern of subscribing to one event and emitting another. Each chaining call creates static DAG edges at registration time -- the graph knows about the A → B connection before any events fire. This makes chaining both a concise wiring syntax and a static declaration of dataflow.

Simple Forwarding

Forward every event from A to B with no transformation.

typescript 
const A = engine.event<number>('A')
const B = engine.event<number>('B')

// Every A automatically becomes a B
engine.on(A).emit(B)

Transform

Apply a transformation function as the payload passes through.

typescript 
const RawTemp = engine.event<number>('RawTemp')
const Celsius = engine.event<number>('Celsius')

engine.on(RawTemp).emit(Celsius, (kelvin) => kelvin - 273.15)

Multiple Outputs

Chain multiple .emit() calls to fan out one event into several.

typescript 
const Input = engine.event<number>('Input')
const Doubled = engine.event<number>('Doubled')
const Squared = engine.event<number>('Squared')

engine.on(Input)
  .emit(Doubled, (n) => n * 2)
  .emit(Squared, (n) => n * n)

Conditional with Skip

Return Skip from the transform function to suppress the emission for certain values.

typescript 
import { Skip } from '@pulse/core'

const Input = engine.event<number>('Input')
const PositiveOnly = engine.event<number>('PositiveOnly')

engine.on(Input).emit(PositiveOnly, (n) => n > 0 ? n : Skip)

Join Chaining

Combine join patterns with chaining. The handler fires only when all input events are ready.

typescript 
const A = engine.event<number>('A')
const B = engine.event<number>('B')
const Sum = engine.event<number>('Sum')

engine.on([A, B]).emit(Sum, ([a, b]) => a + b)

Dispose

Chaining returns a Rule object with a dispose() method to remove the wiring.

typescript 
const rule = engine.on(A).emit(B)

// Later, remove the wiring
rule.dispose()

When to Use Chaining vs Handler

Use Chaining Use Handler
Simple A-to-B forwarding Complex logic with conditionals / loops
Pure transforms (map, filter) Side effects (DOM updates, logging)
Declarative wiring diagrams Async operations (fetch, timers)
Fan-out to multiple targets Stateful computations with closures

Patterns

The API is 3 methods. Everything you used to need dedicated operators for is now a pattern inside on().

State Management

Events + on() + emit a Changed event.

typescript 
const Increment = engine.event<number>('Increment')
const Decrement = engine.event<number>('Decrement')
const CountChanged = engine.event<number>('CountChanged')

let count = 0

engine.on(Increment, (n) => {
  count += n
  engine.emit(CountChanged, count)
})

engine.on(Decrement, (n) => {
  count -= n
  engine.emit(CountChanged, count)
})

engine.on(CountChanged, (value) => {
  document.getElementById('counter')!.textContent = String(value)
})

Animation

Frame event + requestAnimationFrame + spring/tween math inside on().

typescript 
const Frame = engine.event<{ time: number; dt: number }>('Frame')
const PositionChanged = engine.event<number>('PositionChanged')

let position = 0
let velocity = 0
const target = 300
const stiffness = 170
const damping = 26

engine.on(Frame, ({ dt }) => {
  const dtSec = dt / 1000
  const displacement = position - target
  const springForce = -stiffness * displacement
  const dampingForce = -damping * velocity
  velocity += (springForce + dampingForce) * dtSec
  position += velocity * dtSec
  engine.emit(PositionChanged, position)
})

engine.on(PositionChanged, (pos) => {
  element.style.transform = `translateX(${pos}px)`
})

// Drive the frame loop
let lastTime = 0
function loop(time: number) {
  const dt = lastTime ? time - lastTime : 0
  lastTime = time
  engine.emit(Frame, { time, dt })
  requestAnimationFrame(loop)
}
requestAnimationFrame(loop)

Async

on() with an async handler, emit Loading/Done/Error events.

typescript 
const FetchUser = engine.event<string>('FetchUser')
const UserLoading = engine.event('UserLoading')
const UserDone = engine.event<User>('UserDone')
const UserError = engine.event<Error>('UserError')

engine.on(FetchUser, async (userId) => {
  engine.emit(UserLoading, undefined)
  try {
    const res = await fetch(`/api/users/${userId}`)
    engine.emit(UserDone, await res.json())
  } catch (err) {
    engine.emit(UserError, err as Error)
  }
})

Debounce

setTimeout inside a handler.

typescript 
const RawInput = engine.event<string>('RawInput')
const DebouncedInput = engine.event<string>('DebouncedInput')

let timer: ReturnType<typeof setTimeout>

engine.on(RawInput, (value) => {
  clearTimeout(timer)
  timer = setTimeout(() => {
    engine.emit(DebouncedInput, value)
  }, 300)
})

Computed / Derived

on(SourceChanged) -> compute -> emit(DerivedChanged).

typescript 
const FirstNameChanged = engine.event<string>('FirstNameChanged')
const LastNameChanged = engine.event<string>('LastNameChanged')
const FullNameChanged = engine.event<string>('FullNameChanged')

let firstName = ''
let lastName = ''

engine.on(FirstNameChanged, (name) => {
  firstName = name
  engine.emit(FullNameChanged, `${firstName} ${lastName}`.trim())
})

engine.on(LastNameChanged, (name) => {
  lastName = name
  engine.emit(FullNameChanged, `${firstName} ${lastName}`.trim())
})

Conditional

Return early or use Skip.

typescript 
const Input = engine.event<number>('Input')
const EvenOnly = engine.event<number>('EvenOnly')
const OddOnly = engine.event<number>('OddOnly')

engine.on(Input, (n) => {
  if (n % 2 === 0) {
    engine.emit(EvenOnly, n)
  } else {
    engine.emit(OddOnly, n)
  }
})

Performance Leadership

Pulse is not a raw event emitter. Every emit() call traverses a DAG with topological sorting, ref-counted mailboxes, object pooling, and join pattern resolution -- yet it is the fastest library tested. Through fast-path dispatch, solo-consumer caching, and inline join resolution, Pulse turns architectural overhead into a net performance advantage:

  • DAG with topological sorting -- Pulse runs Kahn's algorithm on every propagation cycle to ensure glitch-free, deterministic ordering.
  • Ref-counted mailbox system -- Events are persistent across multiple consumers. Each event tracks which handlers still need to consume it.
  • Object pool recycling -- Events are pooled and recycled (512-entry pool) for GC friendliness. The pool is fast enough to contribute to overall throughput.
  • Join pattern checking -- For join handlers (on([A, B])), the engine checks barrier conditions on every propagation pass.

These features provide: glitch-free ordering, deterministic execution, join patterns, and event persistence. EventEmitter3 and raw RxJS Subjects have none of this -- they are fire-and-forget callback dispatchers. Despite doing fundamentally more work on every emit, Pulse beats them both. At 27.30M ops/sec, Pulse outperforms EventEmitter3 (25.47M) -- a library whose only job is firing callbacks -- by 7%. This is extraordinary for an engine with DAG ordering, mailboxes, and join patterns.

Where Pulse Wins

  • Pulse is #1 in 5 of 7 categories: throughput, chain propagation, fan-out, conditional routing, and deep chain (tied with RxJS).
  • Pulse beats EventEmitter3 in throughput (1.07x), chain propagation (1.76x), fan-out (1.92x), and deep chain (1.35x).
  • Pulse beats RxJS in every category except join (97% -- virtual tie) and deep chain (tied).
  • Fan-out: 4.6x faster than RxJS, 1.9x faster than EventEmitter3.
  • Chain propagation (handler): 1.42x faster than RxJS at 19.17M ops/sec through a 4-level chain.
  • Conditional routing: 1.27x faster than RxJS at 18.55M ops/sec.
  • Memory: 17.7x less heap usage than RxJS (13.3 KB vs 235.4 KB).
  • Pulse beats Zustand in every category (1.94x throughput) and crushes MobX, Jotai, and Redux by 10-34x.
  • Join is the only category where Pulse does not lead, at 97% of RxJS combineLatest -- a virtual tie at 9.68M vs 9.96M ops/sec.

When Does This Matter?

27.3M ops/sec means each emit takes ~29ns. A 60fps frame budget is 16.6ms. You would need to emit ~570,000 events per frame before Pulse's overhead becomes visible. In real-world UI applications, event rates are typically 10-100 per frame. Pulse is not just fast enough -- it is the fastest option available, with glitch-free ordering and join patterns included at no performance cost.
